## short article
In Java, a closure is a block of code (most commonly a lambda expression or an anonymous inner class) that "captures" or "closes over" variables from its surrounding lexical scope. This mechanism allows the function to access those variables even after the method that created them has finished executing. [1, 2, 3, 4] 
### How Closures Work in Java
Java implements closures by bundling the code of a function with its environment—the values or references of variables from the scope where it was declared. [2, 5, 6] 
* Variable Capture: When you create a lambda, Java captures the variables it references from the outer scope. While standard local variables live on the stack and are destroyed when a method returns, captured variables are stored on the heap within the closure object to ensure they remain accessible.
* The "Effectively Final" Rule: To maintain thread safety and predictability, Java requires that any local variable captured by a closure must be final or effectively final. A variable is effectively final if its value is never modified after it is first assigned.
* Implementation: In Java 8+, closures are typically implemented via Lambda Expressions paired with Functional Interfaces. [2, 4, 6, 7, 8, 9, 10, 11, 12, 13] 
### Key Restrictions and Workarounds
Unlike languages like JavaScript or Python, Java has specific limitations on its closures: [14, 15, 16] 
* Immutable Capture: You cannot modify a captured local variable inside a lambda; doing so results in a compile-time error.
* Workaround for Mutability: If you must modify a value inside a closure, use a mutable wrapper object like a single-element array or an AtomicInteger. The reference to the wrapper remains final, while its contents can be changed.
* Static/Instance Variables: These can be modified freely within a closure because they reside on the heap and are not subject to the effectively final restriction of local variables. [7, 9, 10, 17, 18]
