## short article
peer closed connection in SSL handshake (104: Connection reset by peer) while SSL handshaking to upstream

The error "peer closed connection in SSL handshake (104: Connection reset by peer)" occurs when the upstream server (the backend your proxy is trying to reach) abruptly terminates the TCP connection during the initial SSL/TLS negotiation. This typically means the backend rejected the connection attempt before a secure channel could be established.

Common causes and solutions include:
- Missing SNI (Server Name Indication): The backend may require the hostname during the handshake to present the correct certificate.
  - **Fix**: Add proxy_ssl_server_name on; to your NGINX configuration within the location or upstream block.
- Protocol or Cipher Mismatch: The proxy and upstream server might not support the same TLS versions (e.g., proxy tries TLS 1.3, but backend only supports TLS 1.2) or cipher suites.
  - **Fix**: Explicitly define supported protocols using proxy_ssl_protocols TLSv1.2 TLSv1.3;.
- Certificate Trust Issues: The upstream server may require a client certificate for mutual TLS (mTLS), or the proxy is failing to verify the backend's certificate.
  - **Fix**: If the backend uses a self-signed certificate, you may need to disable verification for testing with proxy_ssl_verify off;.
- Incorrect Port or Protocol: You might be trying to connect via HTTPS to a port that only accepts plain HTTP, or vice versa.
  - **Fix**: Verify the proxy_pass destination uses the correct scheme (e.g., https://backend:443 vs http://backend:80).
- Infrastructure Interference: Firewalls, load balancers, or deep packet inspection (DPI) tools between the proxy and upstream may be dropping the connection if they perceive the handshake as invalid.
- System Clock Desync: If the system time on either the proxy or the upstream is significantly incorrect, the SSL certificate may be viewed as not yet valid or already expired.
